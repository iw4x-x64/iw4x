Note that we are intentionally passing `-Wl,--allow-multiple-definition` to
the linker to bypass a known issue in libstdc++.

The background here is that C++26 introduces constexpr support for exceptions.
When compiling with -std=c++26, the __cpp_lib_constexpr_exceptions macro is
defined. To support this, GCC updated <stdexcept> so that the copy constructor
and copy assignment operator for std::runtime_error are explicitly defaulted.
Naturally, defaulting these functions in the class definition makes them
implicitly inline.

The problem is that, for ABI stability and noexcept guarantees, libstdc++
internally backs standard exceptions with a legacy copy-on-write (COW) string.
The implementations for these operations live in cow-stdexcept.cc and are
compiled into cow-stdexcept.o inside libstdc++.a. Currently, GCC 16 fails to
omit or properly mark these out-of-line definitions when the inline defaulted
definitions are active via the C++26 path.

In other words, the linker sees two strong symbols: the inline definition
generated by the compiler from the header, and the out-of-line definition
provided by the archive.

So what we do is pass --allow-multiple-definition to force the linker to
silently accept the first symbol it resolves and discard the duplicate.
Because both the inline and out-of-line definitions perform the exact same
underlying COW copy operations, this workaround is functionally safe and
does not break the ABI.

Now, obviously, throwing a flag like --allow-multiple-definition into our build
have the potential to backfire spectacularly on us. For example, if we ever
accidentally introduce a legitimate multiple definition bug somewhere in our own
codebase, the linker will just silently pick one and swallow the error. But
unfortunately, we don't really have much of a choice here.

Once GCC patches cow-stdexcept.cc to respect the C++26 constexpr exceptions
code path, this flag should be removed.
